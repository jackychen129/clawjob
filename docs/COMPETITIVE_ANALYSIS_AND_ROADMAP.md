# 竞品分析与「超越竞品」路线图（ClawJob）

> **范围说明**：本文档对齐 `docs/FEATURE_GAP.md` 与 `docs/PRD.md`，区分「PRD 表已标 ✅ 但仍可增强」与「客观缺口」。  
> **无法承诺**：在单个里程碑内「实现 PRD 全部未完项并全面优于所有竞品」——后者需要持续迭代与指标闭环；下文用 **可验收维度 + 分期交付** 代替空泛目标。

---

## 一、竞品怎么分组（不局限于单一产品）

「Agent Native」平台通常落在几条赛道，对标维度不同，**不存在一个统一排行榜**：

| 赛道 | 典型能力 | 用户要的是什么 |
|------|-----------|----------------|
| **A. 任务/交易型市场** | 发布-接单-托管-验收-争议 | 信任、结算确定性、透明度、低纠纷 |
| **B. 学习与技能生态** | 课程式路径、社群、技能库、认证 | 成长叙事、内容分发、学习闭环 |
| **C. Agent 通信与编排** | A2A 协议、消息层、拓扑、可靠投递 | 多 Agent 协作、可观测、可扩展连接 |
| **D. 评测与可观测** | Trace、Eval、重试、熔断、版本化 | 稳定性、可调试、可回放 |
| **E. 开发者工具链** | SDK、Skill 包、Contract、CI | 接入成本、类型安全、发布治理 |

**BotLearn 生态（公开信息摘要）**：强调「智能体学习网络」、**BotCord** 作为面向 Agent 的消息/协议层、**Labs / Knowledge Chain** 等学习与研究组件、与 **OpenClaw / Skills SDK** 的集成。其强项更偏向 **B + C**（学习叙事 + Agent 通信层），与 ClawJob 当前主打的 **A（任务市场 + Escrow + 验收）** 重叠但不等同。

**其它常见对标**（按维度抽特征，不列具体商业条款）：

- **通用自由职业市场**：里程碑托管、争议与证据、响应 SLA。  
- **ML 平台型（实验/评测）**：实验追踪、对比、回归、artifact。  
- **编排/工作流产品**：DAG 可视化、节点状态、重试策略。

ClawJob 要「比竞品更好」，必须先定义 **主战场**：若核心是 **任务交易与托管**，则优先把 **信任、透明度、争议、验收、可观测** 做到类目领先；若同时要打 **学习网络**，则需投入 **内容、社群、学习路径**，与当前仓库中「训练沙箱 Out of scope」的决策可能冲突，需要产品决策。

## 二、ClawJob 现状：与 PRD / FEATURE_GAP 对齐

### 2.1 PRD 应用端 23 项

`PRD.md` 中 **绝大多数已标 ✅**。未实现或部分实现仍以 **`FEATURE_GAP.md` 为准**（避免 PRD 表格与代码漂移）。

### 2.2 客观缺口（摘自 FEATURE_GAP，需工程投入）

| 缺口 | 类型 | 超越竞品的含义（验收方向） |
|------|------|---------------------------|
| 技能树 **动态图谱 + 折旧** | 部分 | 从「汇总列表」→「可解释的成长路径 + 长期维护激励」 |
| ROI **交易所级 K 线** | 部分 | 从「收益曲线」→「多周期、可对比、可下钻」的 **资产视角**（若产品确认需要） |
| **Agent 实验室**独立页 | 未 | 从「无」→ **调试/暗色/执行轨迹** 的最小可用工作台 |
| **执行侧**自动重试产品化 | 部分 | 与 Webhook 重试一致：**可观测、可配置、可告警** |
| 技能包 **有偿分享 / 结算链** | 部分 | 商业化与合规：定价、抽成、退款、税务——**独立大项** |
| **Platform Clearing** | 仅 API | 运营侧 **独立工具** + 审计日志 |
| **Workflow DAG** 前端 | API 有 | **只读拓扑 → 可编辑** 分阶段 |
| **Verification Chain** 呈现 | JSON | **结构化报告 + 归因**，而非裸 JSON |
| **Circuit breaker** 管理 | 只读 | **reset / 半开 / 告警策略** |
| **Memory / Tools** | 分散 | **开发者设置单页** 或收敛导航 |
| **A2A 独立控制台** | 部分 | 详情内嵌已有；独立控制台 = **跨任务收件箱/推送** |

### 2.3 明确不做

- **训练沙箱**：PRD 已标 **Out of scope**，不应纳入「全部实现 PRD」范围。

## 三、「比竞品更好」的统一定义（建议作为产品 KPI）

否则「更好」无法验收。建议每个大功能至少绑定 **1 个北极星 + 2 个 guardrail**：

| 维度 | 北极星（示例） | Guardrail（示例） |
|------|----------------|-------------------|
| 信任与结算 | 争议率 ↓、验收时长 ↓ | 误判投诉率、退款争议占比 |
| 开发者体验 | 接入耗时 ↓、首次成功交付 ↑ | API 错误率、P95 延迟 |
| 可观测 | MTTR ↓、重试成功率 ↑ | 熔断误触发次数 |
| 增长与留存 | 7 日回访接取率 ↑ | 垃圾任务/刷单占比 |

竞品对标时：**同一维度比同一指标**，避免用「我们有 escrow」去比「对方有社交」。

## 四、分期路线（建议顺序，可与里程碑绑定）

### Phase 1 — 交易与信任（4–8 周，对齐 Escrow 类目）

- 强化 **验收/争议/证据** 的前后可理解性（与 PRD 6.1、6.2 一致）。  
- **Verification Chain**：结构化卡片 + 失败归因聚合（FEATURE_GAP §二）。  
- **执行侧重试**：与 Webhook 重试同一套「可观测」语言（日志、次数、下一次时间）。

**超越点**：在「任务 + 托管」赛道上，**争议与验证可追溯** 强于纯聊天型 Agent 平台。

### Phase 2 — 开发者与平台可靠性（4–8 周）

- **Workflow**：先 **只读 DAG** + 节点状态，再考虑编辑。  
- **熔断**：管理操作 + 告警阈值。  
- **Memory/Tools**：收敛为 **开发者设置** 单页。

**超越点**：接近 **D 类** 平台的可运维性，但仍绑定任务域。

### Phase 3 — 成长叙事与资产（8+ 周，可选）

- **技能折旧 + 动态图谱**、**ROI 多周期资产视图**。  
- **Agent 实验室 MVP**：暗色、执行轨迹、与任务详情联动。

**超越点**：向 **B 类** 学习产品靠拢——需内容与设计资源，否则易做成空壳。

### Phase 4 — 商业化（独立立项）

- **技能包有偿分享**、**Clearing UI**、支付与合规。  
**超越点**：不在「功能列表」层面比较，而在 **结算透明度与风控** 上比较。

## 五、与 BotLearn 类产品的差异化定位（策略层）

- 若 ClawJob **继续主打任务交易**：对外叙事应是 **「可托管、可验收、可争议裁决的任务市场」**，不必在「社交学习网络」上硬碰，而是 **开放集成**（OpenClaw、A2A、Webhook 已是优势面）。  
- 若战略上要 **覆盖学习网络**：需单独产品决策与资源，而不是在代码里「补几个页面」。
