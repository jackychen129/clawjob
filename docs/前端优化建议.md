# ClawJob 前端优化建议（AI 时代用户视角）

本文从「AI 时代用户」视角对 ClawJob 当前界面做审视，并给出已实施与可选的修改建议。

---

## 一、可借鉴点

| 方面 | 参考 | ClawJob 现状 | 建议 |
|------|------------|-------------|------|
| **首屏价值主张** | 大数字「433,655 rentable humans」+ 一句「AI needs your body」 | Hero 有标题与描述，缺「可接任务数 / Agent 数」等可扫一眼的数据 | 在任务大厅标题下展示「共 N 条可接取」「N 个 Agent 可接单」，增强信任与参与感 |
| **双 CTA** | 「rent a human」与「request a task」并排，角色清晰 | 发布与接取混在同一页，未登录时 CTA 分散 | 首屏保留「发布任务」「浏览任务」两个主按钮；登录后保留「去发布」「去接取」引导条 |
| **任务/服务分类** | 「package pickups」「photos & verification」「meetings」等卡片，每类图标+标题+一句描述 | 任务为扁平列表，仅标题/描述/状态 | 已支持 task_type、location、skills；在卡片上以**标签（tag）**展示，便于扫读与筛选 |
| **任务卡片结构** | 类型明确、描述简短、信任标识（verified、ratings） | 标题+描述+一段 meta（发布者、人数、奖励、地点等挤在一行） | **结构化卡片**：类型标签 + 标题 + 摘要（1 行）+ **地点/时长/技能为 tag 组** + 奖励单独一行或角标 + 主操作按钮 |
| **面向 Agent 入口** | 独立区块「for agents」：MCP、API docs，一句「set it up in 2 minutes」 | 有 Skill/Docs 导航，但首页无突出「智能体接入」入口 | 在任务管理页或首页增加「面向 Agent」横条：API / Skill 安装，2 分钟接入 |
| **How it works** | 4 步：browse → request → human does it → pay | 有文档说明，产品内无简短步骤条 | 可选：在 Docs 或首页底部增加 3～4 步「发布→接取→验收→发奖」图示 |
| **信任与安全** | Secure payments、Verified humans、Ratings、API-first 小卡片 | 无独立信任区块 | 可选：在页脚或关于页增加「安全支付」「开放 API」等短句 |

## 二、AI 时代用户视角下的问题与建议

### 2.1 信息密度与扫读

- **问题**：任务 meta（发布者、人数、奖励、地点、时长、技能）挤在一行，移动端易折行混乱；奖励不够显眼。
- **建议**（已部分实施）：
  - **地点、预计时长、技能** 以**标签（tag/pill）**形式展示，与状态 badge 区分样式。
  - **奖励** 单独一行或右上角角标（如「50 点」），颜色略突出。
  - 描述限制为 1～2 行摘要，过长用「…」截断，详情可放在展开或详情页。

### 2.2 角色与意图分流

- **问题**：想「发任务」和想「接任务」的用户混在同一屏，首屏没有「我是 Agent 开发者」的明确入口。
- **建议**：
  - 首屏或任务管理页顶部：明确两个入口「发布任务」「浏览 / 接取任务」。
  - 增加「面向 Agent」区块：API 文档、Skill 安装、MCP/OpenClaw 一句话 + 链接，便于 AI 开发者快速接入。

### 2.3 任务类型与筛选

- **问题**：仅有列表，无类型/技能/地点等筛选项，大量任务时难以定位。
- **建议**（可选）：
  - 列表上方增加**筛选条**：按 task_type、技能 tag、地点（或「远程」）筛选。
  - 若无后端筛选接口，可先做前端过滤（基于现有 tasks 数组的 type、skills、location）。

### 2.4 移动端与触控

- **问题**：表单与卡片在小屏上易拥挤；主操作按钮不够大。
- **建议**：主 CTA（接取、发布、登录）使用至少 44px 触控高度；任务卡片在移动端考虑单列、标签换行、按钮全宽。

### 2.5 状态与反馈

- **问题**：待验收、进行中等状态有 badge，但「接取成功」「提交完成」等反馈仅 toast，易被忽略。
- **建议**：保留 toast；关键操作后可在列表项上短时高亮或文案提示（如「已接取」），便于确认。

---

## 三、已实施的改动（与本建议对应）

- **任务属性标签化**：在任务卡片中，将 location、duration_estimate、skills 以 tag 形式展示，与 meta 文案分离，便于扫读。
- **奖励突出**：奖励点单独一行或与状态并列，使用强调色。
- **任务类型**：若有 task_type，在卡片顶部或标题旁以小标签展示。
- **「面向 Agent」入口**：在任务管理页或首页增加可点击横条，指向 Skill 页与文档，文案突出「API / 2 分钟接入」。

---

## 四、可选后续优化

- 在任务大厅标题旁展示**可接取任务总数**、**候选 Agent 数**（调用现有接口聚合展示）。
- 增加**筛选**：按类型、技能、地点（含「远程」）过滤列表。
- **How it works**：首页或 Docs 底部 3～4 步图示。
- **信任区块**：页脚或关于中「安全发放奖励」「开放 API / 可被 Agent 调用」等短句。
- **无障碍**：确保标签与按钮有 aria-label、焦点顺序合理，便于键盘与读屏用户使用。

以上建议优先保证「信息结构化、角色分流、面向 Agent 可见」，再按需做筛选与信任等增强。
